## AWS Well Architected フレームワーク 6つの柱

### 運用上の優秀性
- AWSで構築したシステムがビジネスにおいて価値を提供し続けるために、運用面でどのような点に考慮すべきか


### セキュリティ
- AWS上で構築したシステムとデータをどのように保護するのか


### 信頼性
- 障害の防止、障害の復旧


### パフォーマンス効率
- コンピューティングリソースの効率的な使用


### コスト最適化
- 不要なコストの回避や最適化


### 持続可能性
- エネルギー消費量を削減し、地球環境への影響を最小限にするために必要な設計、実装


## AWS責任共有モデル
- クラウド事業者であるAWSと利用者であるユーザーとがセキュリティ対策の責任境界を明確にすることで、分担・協力しながらセキュリティを強化していくという方針をとっており、これをAWS責任共有モデルと呼ぶ


| 利用者の責任範囲 | 利用者のデータ |
| --- | --- |
| クラウド内のセキュリティ | プラットフォーム、アプリケーション、IDとアクセス管理 |
|  | OS、ネットワーク、ファイアウォール構成 |
|  | クライアント側のデータ暗号化とデータ整合性認証、サーバー側の暗号化（ファイルシステムやデータ）、ネットワークトラフィック保護（暗号化、整合性、ID） |
|  |  |
| AWSの責任範囲 | ソフトウェア |
| クラウドのセキリュティ | コンピュート、ストレージ、データベース、ネットワーキング |
|  | ハードウェア/AWSグローバルインフラストラクチャ |
|  | リージョン/AZ/エッジロケーション |

- AWSの責任範囲については上記の表以上に語ることはなさそう

- 利用者の責任範囲（クラウド内のセキュリティ）
  - 必要最低限の権限をユーザーに付与する最小権限の原則の考え方に従う
  - VPCのセグメンテーション、セキュリティグループやネットワークACl、Webアプリケーション保護
  - S3などのAWSサービス側で暗号化するSSE、クライアント側で暗号化するCSE、AWS上で暗号鍵を管理するAmazon Key Management Service(KMS)
  - セキュリティ監視


## AWSグローバルインフラストラクチャ

### リージョン
- 地理的に離れたある領域のこと 2022年時点では30箇所くらいあるそう

- ap-northeast-1, 東京, シンガポールとかそういうの


### アベイラビリティゾーン(AZ)
- リージョン内にある複数のデータセンターなどの集合体 2022年時点では96箇所


### マルチAZ
- 上記のAZを複数個使用してシステムを構成する、マルチAZ構成という文脈で使う

- 高い可用性や災害復旧（DR）サイトを構築する場合に選択される


### マルチリージョン
- 可用性を高めるために、地理的に離れた場所にもシステムを構築する際に使われる

- 語義としては、単に複数のリージョンというだけ。ただ複数のリージョンを用いるからには相当の可用性などが要求されたり、その地域から配信してレスポンスの向上とか目指したりなのかな


### エッジロケーション
- CDNなどのAZを上回る数が各地に用意されているグローバルインフラストラクチャサービスを提供するための拠点

- ユーザーは地理的に近い拠点にアクセスすることでレスポンス速度の向上が期待できる


## セキュリティ・アクセス管理

### IAM(AWS Identity and Access Management)
- AWSを利用するユーザーに対してAWSへのアクセスを安全に制御するための仕組み

- 最初に登録しメールアドレスでAWSの操作の入り口となるAWSマネジメントコンソールにログインできる（ルートユーザー 管理者権限が付与されるのでシステム運用では使わない）

- IAMユーザーおよびIAMグループを作成し、実際に利用するユーザーやグループを登録できる

- IAMポリシーをIAMユーザーまたはIAMグループに対して、適切な権限を付与する

  - Webブラウザからマネジメントコンソールにログインする
  - AWS CLIでWindowsやLinuxからコマンド動作する CLI操作にはアクセスキーIDとシークレットアクセスキーを事前に作成しておく必要がある
  - AWS SDKでプログラムからAPIを利用する

  - 認証したユーザーごとに一時的なアクセストークンを発行することで認証連携を実現している(Security Token ServiceでTemporary Security Tokenを発行する)


### IAMユーザー
- AWSを操作するユーザーをIAMサービスから作成

- 1アカウントで5000ユーザーまで作成可能

- IAMユーザーには後述のIAMポリシーで権限を直接割り当てることもできる

- しかし、IAMグループに所属させ、IAMグループに権限を付与すると管理しやすい

- AWSマネジメントコンソールを利用する際には、IAMユーザー名とパスワードの認証情報を使用する

- アクセスキーIDとシークレットアクセスキーのペアを作成することができるが、現在は推奨されていない


### IAMグループ
- IAMユーザーを束ねるグループを作成することで、ユーザーや権限などの管理を統合的に行うことができる

- 1アカウントで最大300グループまで作成でき、IAMユーザーあたり10までのグループに所属できる


### IAMポリシー
- 新規に作成されたばかりのユーザーやグループは最初は何も権限が付与されておらず、最低限必要となる権限のみをIAMポリシーを利用してIAMユーザーやIAMグループに付与する


```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Action": [
        "s3:GetObject"
      ],
      "Effect": "Allow",
      "Principal": "*",
      "Resource": "arn:aws:s3:::test-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": [
            "11.22.33.44/32"
          ]
        }
      }
    }
  ]
}
```

  - 明示的な拒否 → 明示的な許可 → 暗示的な拒否

- IAMポリシーの主な種類
  - **AdministratorAccess** すべてのアクセス権限を提供するポリシー。AWSアカウントを利用しない代わりに、管理者相当のIAMユーザーやIAMグループに付与する
  - **PowerUserAccess** IAMサービスを除くその他すべてのアクセス権限を提供するポリシー。通常業務ではユーザー管理やポリシー管理を非有長としないIAMユーザーやIAMグループに付与する
  - **カスタム管理ポリシー** AWSのユーザーが自身で作成・カスタマイズできるポリシー。標準提供されているAWS管理ポリシーではセキュリティ要件を満たせない場合に使用


### IAMロール
- AWSサービスやアプリケーションにAWSの操作権限を付与する仕組み

- IAMユーザーやIAMグループと紐付けせず、IAMロールに対してIAMポリシーを直接付与することで権限を管理する

  - AWSサービスだけでなく、外部WEBアプリケーションなどにも設定できる
  - 外部サービスからAssumeRoleをリクエストし、IAMが一時的な認証情報を発行することで、外部サービスがAWSリソースにアクセス可能になる


### AWS Organizations
- 1つの組織内での複数のアカウントに対して統合的なアカウント管理を行えるサービス

- 使用量を一括請求にすることも可能


### サービスコントロールポリシー(SCP)
- OrganizationsではSCPという重要な機能が提供されており、複数のAWSアカウントに対して、IAMポリシーのような権限制御を統合的に管理・運用する機能

- 組織内のIAMユーザーやIAMロールが使用できる最大権限を一元的に制御する

- SCP自体は権限を付与しない

- IAMポリシーで許可されていても、SCPで許可されていない操作は実行できなくなる

- 組織内のメンバーアカウントにのみ影響する

- 管理アカウントのユーザーやロールには影響を与えない、ルートユーザーにも適用されない

- SCPとIAMの組み合わせ
  - SCPで許可 + IAMで許可 → アクション実行可能
  - SCPで拒否 + IAMで許可 → アクション実行不可
  - SCPで許可 + IAMで許可なし → アクション実行不可


### IAM Identity Center
- 複数のAWSアカウントへのアクセスを一元的に管理することができる

- サードパーティ製のSAML2.0互換IDプロバイダーを用いたIDフェデレーションを実施することが可能


### AWS Managed Microsoft AD
- AWS上でフルマネージド型のMicrosoft Active Directoryを提供するサービス


### AD Connector
- 既存のオンプレミスActive Directoryに接続するためのプロキシサービス


## ネットワークサービス

### VPC (Amazon Virtual Private Cloud)
- 特定のユーザーだけが利用できるネットワーク環境を構築する

- 特定のリージョンに独立したプライベートネットワーク空間を作成できる

  - IPv4のVPCはプライベートIPアドレスの範囲内で、16ビット以上28ビット以内のCIDRを指定する必要がある
  - VPC間を接続したりオンプレミス環境と接続する場合は、IPアドレスの範囲が重複しないように注意する必要がある
  - IPv6のVPCは56ビットのCIDRに固定されている

  - **DNS設定** VPCが構築する仮想ネットワーク内でDNSによる名前解決を適用するための設定が求められる
    - **enableDnsSupport** DNS解決がサポートされているかどうかを示す

  - **VPCエンドポイント（ゲートウェイ型）** VPCに構成されるもので、S3バケットへのルートを設定できる。VPC内外の通信に信頼性の高い接続を提供する


### サブネット
- VPC内に構築するネットワークセグメント

- 1つのVPCに対して複数作成することができるが、VPCのIPアドレス範囲内でCIDRを設定する必要がある

- セキュリティと可用性の観点で構築する必要がある

- サブネットにはパブリックサブネットとプライベートサブネットがあるが、インターネットとの通信が必要か否かで判断すれば良い

- 異なるAZに同じ構成のサブネットを構築し、可用性を確保するということもある


### パブリックサブネット
- インターネットとの通信が可能なサブネット

- インターネットと通信が必要なサーバを配置する


### プライベートサブネット
- パブリックに対して、インターネットとの通信ができないサブネット

- インターネットと通信する必要がないサーバを配置する


### インターネットゲートウェイ(IGW)
- VPC内のリソースからインターネットにアクセスするためのゲートウェイ

- VPCの玄関みたいなイメージ


### ルートテーブル
- サブネット内のリソースに対する静的ルーティングを定義する

- サブネット単位で行う

- ルートテーブル内のデフォルトゲートウェイ(0.0.0.0/0)へのルーティングにインターネットゲートウェイを指定したサブネットがパブリックサブネット

- ルートテーブル内のデフォルトゲートウェイ(0.0.0.0/0)へのルーティングにインターネットゲートウェイを指定していないのがプライベートサブネット


### NATゲートウェイ
- プライベートサブネットからネットワークアドレス変換機構を提供してくれる

- NATゲートウェイはパブリックサブネットに設置する


### NATインスタンス
- NATゲートウェイを用いない場合にNAT機能を持ったEC2インスタンス


### セキュリティグループ
- EC2インスタンスなどに適用するファイアウォール機能、EC2インスタンスから出る通信を制御するアウトバウンド、EC2インスタンスに入る通信の制御をするインバウンドの2つを設定する

- デフォルトの設定ではアウトバウンドは全て許可、インバウンドは全て拒否

- インバウンド/アウトバウンドの種別、プロトコル(TCPやUDP)、ポート範囲、アクセス元またはアクセス先IDなどを許可するルールのみ定義

- 複数のセキュリティグループ(セキュリティグループの集合体)を適用可

- セキュリティグループの設定追加・変更は即座に反映される

- ステートフルな制御が可能(ルールで許可された通信の戻りの通信も自動的に許可される)


### ネットワークアクセスコントロールリスト(ACL/NACL)
- サブネット単位で設定するファイアウォール機能

- セキュリティグループと同様にインバウンドとアウトバウンドの制御が可能で、異なるサブネット間の通信を制御する意図で使われる

- VPC内に構成したサブネットごとに1つのネットワークACLが設定可能

- VPC作成時にデフォルトのACLが用意されており、デフォルトでは全て許可するようになっている

- 新規にネットワークACL(カスタムネットワークACL)を作成することができ、その場合は全てのトラフィックを拒否

- インバウンドとアウトバウンドに対してそれぞれを明示的に制御可能

- ステートレス セキュリティグループとは異なり、インバウンドとアウトバウンドの設定が必要

  - ルール番号の低い順に評価され、最初の合致が優先
  - 合致したルールが正に反映される


### AWS Web Application Firewall(WAF)
- 送信元のIPアドレスに基づくアクセス制限やHTTPプロトコル情報（HTTPヘッダや本文、URI文字列など）に対してフィルタリングを設定し、特定の攻撃パターンからWebアプリケーションを保護できる
- CloudFrontやALB、API Gatewayなどにアタッチして各サービスへのアクセスを制限することができる

- フィルタリング条件
  - リクエストのIPアドレスの発信元
  - リクエストの発信元国
  - リクエスト内の文字列一致または正規表現一致
  - リクエストの特定部分のサイズ
  - 悪意のあるSQLコードやスクリプトの検出


### AWS Shield
- DoSやDDosなどに代表される一斉攻撃を防御するにはAWS Shieldが適している

- 無償版と有償版がある

  - Advanced版は有料サービスなのでDDoS攻撃によって生じる可能性のあるコストを保証するといった追加機能が提供されている


### AWS Firewall Manager
- SGをはじめ、WAFやShieldやNetwork Firewallなどのルール設定を一元管理できるサービス


### AWS Direct Connect
- オンプレミス環境とAWSの間を専用線で接続するサービス

- DXと呼ばれることもある

- 追加機能としてDirect Connect ゲートウェイというのもあり、オンプレミス環境とVPCを1対1でしか接続できなかったものが、全リージョン内の複数のVPCと1対多で接続可能になる（逆に単にDirectConnectは1対1ということ）

- AWS Site-to-Site VPNと比較して高いスループットや安定性を誇る反面、導入まで時間を要したり、コストが高くなる


### AWS VPN(Virtual Private Network)
- オンプレミスネットワークやクライアントデバイスとAWSネットワーク間をVPN接続するサービス

- AWS Site-to-Site VPNとAWS Client VPNで構成されている


### AWS Site-to-Site VPN
- Direct Connect以外にもIPSecを用いてセキュアに通信できるVPNサービス

- インターネットを利用しているためスループットや品質は低下するが、コストや短期間で導入できることがメリット

- AWS Site-to-Site VPNのトンネルは最大1.25Gbps(条件によっては上振れることも)


### AWS Client VPN
- クライアントデバイスからEC2などのVPC内のリソースへのVPN接続を提供する

- フルマネージド型のサービスのため、従来のようにVPNサーバを構築せずにVPN環境を提供することが可能


### VPCピアリング
- 異なるVPC間をプライベート接続するためのサービス


### AWS Transit Gateway
- VPC内にハブの機能を備えたゲートウェイを配置するサービス


### VPC エンドポイント
- VPC内のリソースから各種AWSのサービスへのアクセスやAPIコールをインターネット経由せずにプライベート接続で可能にするサービス

- ネットワークレイヤーのゲートウェイ型とアプリケーションレイヤーのインターフェイス型がある

- ゲートウェイ型はS3やDynamoDBなどのプライベートIPアドレスをルートテーブルに追加することで可能、無料で接続できる

- インターフェイス型はCloudWatchやSQSなどのAPIコールが可能になる、料金が発生する


### AWS PrivateLink
- インターネットを介さずにAWSサービスや他社のサービスにプライベート接続を提供する

- Direct Connectはオンプレミス環境とVPC間の専用線接続、Site-to-Site VPNはオンプレミス環境とVPC間のVPN接続を構築で、どちらもオンプレミス環境とVPC間の接続というところがPrivateLinkと違う


### Elastic Load Balancing(ELB)
- EC2や特定のIPアドレスへのトラフィックを分散するロードバランシングサービス

- EC2インスタンスを登録するだけで使える、ここで登録されたEC2インスタンスをバックエンドインスタンスと呼ぶ

- SSL/TLSによる通信の暗号化をサポートしている

- ELBの種類
  - **CLB (Classic Load Balancer)** 標準的なロードバランシングを提供し、TCPリクエストをバックエンドインスタンスに振り分け、指定されたポートへトラフィックを転送する
  - **ALB (Application Load Balancer)** リクエストレベル(レイヤー7)で動作しリクエストの内容に応じて、あらかじめ設定したターゲットにルーティングして処理を振り分ける。そのためWebサーバやアプリケーションサーバなどで利用するHTTPやHTTPSトラフィックを特定のサーバに振り分けることができる
  - **NLB (Network Load Balancer)** レイヤー4で動作し、低レイテンシーで高いスループットを実現する場合に利用、送信元のアドレスを保持するため、レスポンスはクライアントへ直接返す
    - アプリケーションに対するHTTPエラーをNLBで検出してEC2インスタンスを自動的に再起動するためには、ヘルスチェックプロトコルでHTTP(またはHTTPS)に変更し、Auto ScalingグループのELBヘルスチェック機能を有効化して、NPBのHTTPエラーが検出された場合にインスタンスが再開するようにAuto Scalingアクションを構成する
  - **GWLB (Gateway Load Balancer)** レイヤー3で動作し、ファイアウォールや侵入検知などの仮装アプライアンスの可用性を高められる

- ELBの特徴
  - トラフィックを複数のAZヘ分散できるため、また正常なインスタンスのみと通信する
  - トラフィックの負荷に応じて自動でスケーリングする機能を備えている、すなわちELB自体にも冗長性が確保されている
  - ELBはVPCサブネットのIPアドレスが自動的に割り振られているが、通常はエンドポイントと呼ばれるDNS名を使用している
  - SSL復号を備えている
  - バックエンドインスタンスが正常に動作しているかヘルスチェックを行い、異常なインスタンスには振り分けを停止する
  - スティッキーセッションという一度セッションを確立したバックエンドインスタンスにユーザーのリクエストを送信できる
  - Connection Drainingという機能を用いるとバックエンドインスタンスをELBから解除する際に、処理が完了するまで接続解除を遅延させることが可能
  - ELBは複数のAZに登録されたすべてのインスタンスに対してリクエストを均等に分散
  - 外部ELB(インターネット公開向け Internet-facing)はELB自体がパブリックサブネットに配置されているので、EC2はプライベートサブネットに配置できる
  - 内部ELB(Internal)はVPC内やオンプレミス環境からのみ利用する

- HTTPトラフィックとHTTPSトラフィックをそれぞれ使ってるけど、すべてのリクエストがHTTPSを利用するためにはHTTPをHTTPSにリダイレクトするリスナールールを設定する


### AWS Certificate Manager(ACM)
- 複数のAWSサービスで利用するSSL証明書を一括で管理することができる

- AWSで利用するSSL証明書を新規で作成も可能だし、既存のSSL証明書もインポートできる

- 有効期限が近づいているACM証明書を検出するEventBridgeルールとしてLambda関数を呼び出すようなものを作成する。SNSによってカスタムアラートを通知するようなLambda関数を設定する。


### Amazon CloudFront
- Amazon CloudFrontはエッジロケーションからコンテンツを配信するCDN(Content Delivery Network)

- オリジン(とはキャッシュするコンテンツの提供元)としてELBやAmazon EC2、Amazon S3を指定できる

- SSL/TLS
  - SSL証明書を利用したSSL暗号化通信が可能
  - ユーザーが独自のSSL証明書を導入可能
  - CloudFrontからバックエンドのAWSサービスまでをSSL暗号化通信にすることも可能

- **署名付きURL** 一定時間だけアクセスを許可するためのURLを発行し限定的なユーザーにのみ公開できる機能
  - 署名付きCookieと比べ個別ファイルへのアクセスを制限する場合に利用する
  - 署名付きURLの方が優先される（この場合はCookieの確認は行われない）

- **署名付きCookie** コンテンツに直接アクセスするURLではなく、署名付きCookieからのアクセスさせる
  - 複数の制限されたファイルなどへのアクセスを提供する場合に利用する
  - 現在のURLを変更したくない場合に利用する

- **カスタムエラーページ** オリジンからエラーコードが返された場合、あらかじめ設定したエラーページを表示することが可能

- **地理的制限** CloudFrontに接続するユーザーの地域情報に基づいてアクセスを許可または拒否できる

- **ストリーミング** Webサービスのコンテンツ配信に加え、映像や音声のストリーミング配信にも対応

- エッジロケーションの価格クラス
  - ALLにするとコストが高い地域も含まれる
  - 制限すると北米や欧州、アジアなどコストが比較的安い地域の主要なエッジロケーションのみを使用するので、コスパが良い

### Amazon Route 53
- 可用性の高いDNSを提供するマネージドサービス

- ホストゾーンの設定として外部向けDNSのパブリックホストゾーンと、VPC内DNSのプライベートホストゾーンがある

  - Zone ApexレコードをAレコードとして扱える

  - 単にDNS機能だけでなく、通信を制御することもできる

- ルーティングポリシーの種類
  - **レイテンシーベースルーティング** レイテンシーが最も低いリソースにルーティングする
  - **加重ルーティング** 複数のリソースに加重度を設定し、指定した比率に応じて処理を分散するようにルーティングする
  - **地理的ルーティング (Geolocation)** ユーザーとリソースの地理的場所に基づいてルーティングする
  - **フェイルオーバールーティング** ルーティング先の対象になるリソースをヘルスチェックし、利用できるリソースにルーティングする
  - **シンプルルーティング** 設定されたレコードの情報にしたがってルーティングする
  - **地理的近接性ルーティング (Geoproximity)** 接続元のクライアントの位置から、地理的に近い場所にルーティングする
  - **複数値回答ルーティング** 最大8個からランダムに選ばれた正常なレコードを使用して、Route 53がDNSクエリに応答する。RDSレスポンスを分散させるため、異常なEC2インスタンスへトラフィックが送られるのを防ぎ、可用性を高めることができる


### Route Origin Authorization(ROA)
- 企業が所有するパブリックにルーティング可能なIPv4またはIPv6アドレス範囲をAWS環境に持ち込み利用する機能はBYOIP(Bring Your Own IP)と呼ばれる

- BYOIPを利用してアドレス範囲をAWSに持ち込む際、AWSはそのIPアドレス範囲がユーザーによって正当に制御されているものかを検証する。この検証と、AWSがそのIPアドレスをインターネット上でアドバタイズすることを承認するために、RIRを通じてRoute Origin Authorizationと呼ばれるオブジェクトを作成する必要がある。ROAを利用して所有権の証明及び経路広告の承認を行うのが正しい移行手続きとなる


## コンピューティングサービス

### EC2(Amazon Elastic Compute Cloud)
- AWSクラウド上で仮想サーバーを提供する、代表的なコンピューティングサービス

- Amazon Machine Image(AMI)の選択 EC2インスタンスはAMIと呼ばれるイメージファイルから起動するイメージファイルにはインスタンスの起動に必要なOSイメージとソフトウェア設定が含まれているため、インスタンス起動後すぐにサーバーとして利用することができる

- インスタンスタイプの選択→サーバースペックのことで、vCPU(仮想CPU)とメモリ容量のバランスによってインスタンスファミリーと呼ばれるグループにわかれている

### ECS(Amazon Elastic Container Service)
- フルマネージド方のコンテナオーケストレーションサービスで、Dockerコンテナを簡単に実行、停止、管理することができる

  - **タスク定義** Dockerイメージの格納先やメモリ使用量の制限、ネットワーク、ストレージ、ヘルスチェックに関する項目の設定をする
  - **ロードバランサー連携** トラフィックをコンテナ全体に分散して処理できるようにする
  - **サービス** クラスター上で起動するコンテナ数や起動するコンテナの最大数を設定する

## ストレージサービス

### S3
- 高耐久・大容量のオブジェクトストレージサービス

  - イレブンナインと呼ばれる高耐久性を備えている。誤ってデータを削除してしまうケースではオブジェクトロック機能やバージョ二ング機能を有効することで対策できる
  - 大規模災害の影響に対して、クロスリージョンレプリケーションで対策できる

- S3のアクセス制御
  - **IAMポリシー** IAMユーザーやIAMロールからRead、Get、Put操作に対する制御などを行う
  - **バケットポリシー** JSONコードを記載してバケット全体のアクセス制御を行う

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "NotPrincipal": {
        "AWS": "arn:aws:iam::111122223333:user/userA"
      },
      "Action": "s3:*",
      "Resource": "arn:aws:s3:::test-bucket"
    }
  ]
}
```

  - **アカウントレベル制御** AWSアカウントレベルのアクセス制御を行う
  - **パブリックアクセスブロック** ブロックパブリックアクセスによるアクセスを行うことで、バケットポリシーやACLの設定に対して一括で制限を設けることができる

  - クロスリージョンレプリケーションを設定することで、バケット内のオブジェクトが作成、更新、または削除される際に発生するイベントをトリガーとして、別のS3バケットにオブジェクトデータのレプリケーションが行われる

- サーバーサイド暗号化 (SSE) の種類
  - **SSE-S3** S3のデフォルトキーを使用したAES-256暗号化
  - **SSE-KMS** AWS Key Management Service(KMS)で管理されている鍵によるAES-256暗号化
  - **SSE-C** ユーザーの任意の鍵によるAES-256暗号化

- 高速アップロード・転送
  - マルチパートアップロードという機能が利用できる
  - Amazon S3 Transfer Accelerationを有効化すると、クライアントとS3間の通信を高速化させることができる

  - PUT/COPY/POST/DELETE 毎秒3500リクエスト以上
  - GET/HEAD 毎秒5500リクエスト以上

- 静的Webホスティング
  - 静的コンテンツのWebサイトホスティング
  - ただし署名付きURLが有効な期間は別ホストからアクセスできてしまうので、真にCloudFront経由でのみ公開したい場合は、WAFのReferer制限をする必要がある

  - ユーザーが管理している鍵
  - Amazon S3で自動生成された鍵

  - 通常はバケット所有者が負担するデータ転送料金とリクエスト料金を、データにアクセスする側に負担させることができる

### Amazon Elastic Block Store(EBS)
- 永続可能なブロックストレージサービスで、EC2インスタンスにアタッチすることで利用できる

- ブロックストレージとは、データをブロックと呼ばれる細かい単位で分割して保管する方式

  - 汎用SSD(General Purpose SSD: gp2およびgp3)
  - プロビジョンドIOPS SSD(PIOPS SSD: io1およびio2)
  - スループット最適化HDD(st1)
  - コールドHDD(sc1)
  - マグネティック

- スナップショットを用いてバックアップを取得することができる

- ボリューム作成時に暗号化を設定できる

  - 既存ボリュームのスナップショットを作成
  - 作成したスナップショットを複製する際に暗号化オプションを指定
  - 暗号化されたスナップショットからEBSボリュームを再作成
  - EC2インスタンスから既存のEBSボリュームをデタッチ
  - EC2インスタンスへ暗号化されたEBSボリュームをアタッチ

    - パフォーマンスの向上、複数のディスクを1台のディスクのように扱い読み書きを高速化する構成
    - ストライピングと呼ぶ
    - ボリュームの冗長性を高める
    - 2つのボリュームを同時にミラーリングする

  - 停止時に削除するか否かの属性

- **Snowmobile** Snowballを超える大容量データをAWSへ移行するケース場合に使用し、100PBのストレージを利用でき、大規模な専用トレーラーを利用して移行される

## データベースサービス

### Amazon Relational Database Service (RDS)
- 対応エンジン
  - Amazon Aurora
  - PostgreSQL
  - MySQL
  - MariaDB
  - Oracle Database
  - Microsoft SQL Server

- インスタンスクラス
  - 汎用: db.m3 (汎用的なワークロード向け)
  - バースト: db.t4g, db.t3, db.t2 (ベースラインのCPUパフォーマンスに加え負荷増大時にバースト可能な性能を発揮)
  - メモリ最適化: db.r6g (メモリを多用するワークロード向け)
  - 最適なパフォーマンスを得るためには、最新世代のインスタンスクラスを使用することが推奨

- インスタンスの役割
  - **プライマリインスタンス** 読み取りおよび書き込みオペレーションをサポートし、データのすべての変更を実施
  - **リードレプリカ** ソースDBインスタンスのデータを非同期で複製する読み取り専用インスタンス。読み取りトラフィックの分散や災害対策(DR)、必要に応じてスタンドアロンのDBインスタンスへ昇格させる用途で使用される
  - **Auroraレプリカ** Amazon Auroraにおけるリーダーインスタンス。プライマリと同じストレージボリュームを共有し、読み取り負荷のオフロードや、プライマリ障害時の自動フェイルオーバー先として機能（最大15個まで作成可能）

- ストレージタイプ
  - 高性能・低レイテンシー向け
  - コスト効率重視、汎用向け
  - 旧世代、非推奨

- バックアップ: 標準では1日1回自動バックアップが実施

- マルチAZ: 複数のAZに構築することを指し、耐久性と可用性が向上する、コストはかかる

- リードレプリカ
  - プライマリデータベースを複製し、読み取り専用として構築したデータベース
  - 異なるリージョンにレプリカを作成することも可能
  - 暗号化にはAWS Key Management Service(KMS)のキーだけでなく、AWSマネージドキーやカスタマーマネージドキーも利用できる

- SSL接続

  - Lambdaなどの複数のクライアントアプリケーションからRDSやAuroraに接続する場合、データベースへのコネクションを都度作成していると最大接続するの上限に達してしまい、接続エラーが発生することがある。RDS Proxyは接続の開閉や保持をアプリケーションに変わって実行する

  - AWSが開発したリレーショナルデータベースサービスで、PostgreSQLやMySQLと互換性がある。性能はPostgreSQLやMySQLよりも優れており、それら以上のスループットを実現できるとされている。
    - クラスターボリュームは３ヶ所のAZに存在し、そのうちの一つはプライマリーデータベスインスタンスと呼ばれ、読み書きを行う
    - 他二つはAuroraレプリカと呼ばれ、読み取りのみを行う
    - 読み書きを全インスタンスでも行う構成のこと

- DBインスタンスのプロセスやスレッドがメモリをどのように活用するかをモニタリングするためには、Amazon RDSの拡張モニタリングを有効化する


### Amazon Aurora Global Database
- 複数リージョンにAuroraクラスターをデプロイすることができる特定の1つのリージョンにプライマリが配置され、それ以外は異なるリージョンに最大５つまでセカンダリDBクラスターを配置することができる

- 書き込みはプライマリークラスタのみで行われ、セカンダリんいはレプリケートされる


### Aurora Serverless
- データが格納されているクラスターボリュームだけが存在し、リクエストを受けて初めてデータベースインスタンスが起動する

- 利用頻度があまり高くない場合や負荷が一定ではないアプリケーションのデータ格納に適している


### Amazon DynamoDB
- マネージド型のNoSQLデータベス

- NoSQLは構造化データ以外の半構造化/非構造化データを扱うことができる


| データ構造の種類 | 説明 | データおよびファイルの例 |
| --- | --- | --- |
| リレーショナルデータ構造化データデータを管理する構造を決め、その構造に合わせて格納したデータ | 企業の期間システムなどで管理されているデータなど | RDB, Excel |
| 非リレーショナルデータ半構造化データ非構造化データを管理できる柔軟性のある構造を用意し、その構造に格納されたデータ | システムのログ、センサー(IoT)データ、位置情報、SNSデータ、設定ファイルなど | LOG, JSON, XML |
| 非リレーショナルデータ非構造化データ構造が定義されておらず、データの関係をモデルに当てはめることができないデータ | テキストファイル(メモ書き)など、音声データなど | TXT, MPEG |

  - 1つのデータはキーとバリューから構成され、データとして書き込む際にはこのセットで保存される、DynamoDBはこれを採用している

  - 自動的に３ヶ所のAZに保存される

  - 書き込んだデータは時間が経てば正しく反映されるが、読み取るタイミングによっては書き込んだデータが反映されない状態になる
  - 強力な整合性を備えた読み込みをすることによって、確実に書き込み後のデータが取得できる

  - レコードの保存期間を設定できる
  - TTL(Time to LIVE)機能を利用して設定し、これを過ぎると自動で削除される

  - RDSと同様にDynamoDBも備わっている。人為的なミスによるデータ書き換えや削除などからデータを保護することができ、過去35日間の任意の時間の時点にデータを戻すことができる

  - デーブル内データのよみかきや有効にしているオプションが課金対象
  - キャパシティモードと呼ばれる設定によって異なるが、2種類をユースケース次第で選択する(読み書き、トラフィックが予想できない場合はオンデマンド、できる場合はプロビジョニング)

| キャパシティモード | 説明 | ユースケース |
| --- | --- | --- |
| オンデマンド | 課金はDBのテーブルに実行したデータの読み書きに対して発生。ワークロードの拡大や縮小に即座に対応できるため、事前の予測や設定は不要 | • 未知のワークロードで新規テーブルを作成し利用する  • DBにアクセストラフィックが予測できない  • 利用した分だけ支払う方が良い |
| プロビジョニング | 事前に一秒あたりの読み書き回数を予測し、Auto Scalingを指定すれば指定した利用率に応じてテーブルのキャパシティが自動的に調整可能 | • DBにアクセスするトラフィックが予測できる  • DBにアクセスするトラフィックが一定、または少しずつ変化する |


- DynamoDBからのS3バケットへのバックアップの取り方には、AWS Glueにより定期的にテーブルデータをS3にエクスポートできる

  - テーブルに保存されたデータ項目に対する追加・変更・削除の発生時にその履歴を保持する機能（24時間以内のデータ変更の履歴を保存）


### Amazon DynamoDB Accelerator(DAX)
- 極めて高いパフォーマンス

- DynamoDBとのAPI互換性

- フルマネージド

- 高いスケーラビリティ

- DAXクラスターの暗号化はデフォルトで有効になっているため、テーブル作成時に無効に変更しない限り暗号化が適用される

- 高コストなので常時使用することは経済的ではない

- ホットパーティションの解決に役立つ


### Amazon Redshift
- ペタバイトクラスのデータを扱うことができるデータウェアハウスサービスで、BIツールなどを利用した大量データの集計、分析に向いてる

- ノードと呼ばれるコンピューティングリソースの集まりで構成されている。リーダノードというノードが処理を受け付け、コンピュータノードに処理を依頼する。これらのノードの集まりをクラスターと呼ぶ

  - 自動で取得するようにしている場合は、8時間ごと。または5GBのデータ変更があった場合に自動で取得される
  - 自動で取得されたスナップショットはRedshiftによって消されるので、永続的にバックアップを保存したい場合は手動でやる
  - 格納先は標準ではRedshiftのクラスターが配置されているリージョンになるが、別のリージョンにもコピーするように設定でき、クロスリージョンスナップショットと呼ぶ

  - プロデューサー側とコンシューマー側に分かれている

- Redshiftクラスターに対するトラフィックを監視するには、拡張VPCルーティングを有効にする

  - ストレージコストの増加を抑えながらもデータをクエリする機能としてRedshift Spectrumがある


### Amazon ElastiCache
- マネージド型のインメモリデータベースで、メモリ上で処理を実行するため、高スループットかつ低レイテンシー

- 真にキャッシュとして使うこともできるし、揮発しても良いデータを保存して計算させるということも可能

- **Valkey / Redis OSS** 高度なデータ構造や高可用性機能が必要な場合に適している
  - 複雑なデータ型: 文字列、ハッシュ、リスト、セット、ソート済みセット、ビットマップ、HyperLogLog、地理空間インデックスなどが必要な場合
  - ソートとランキング: インメモリデータセットのソートやランクづけを行いたい場合
  - 永続性: キーストアの永続化が必要な場合
  - 高可用性とレプリケーション: プライマリから1つ以上のリードレプリカへデータを複製し、読み取り負荷を分散させたい場合、プライマリーノード障害時の自動フェイルオーバーが必要な場合
  - Pub/Sub機能: クライアントへサーバー上のイベントを通知する発行/購読機能
  - バックアップと復元
  - データ階層化
  - Redis Streams
  - ロールベースのアクセスコントロール(RBAC)

- **Memcached** シンプルなキャッシュ用途でマルチスレッド性能が必要な場合
  - スケーリング: 需要の増減に応じてノードを追加・削除する
  - オブジェクトキャッシュ: 単純なオブジェクトのキャッシュとして利用する
  - 逆に複雑なデータ操作、高可用性、永続化、バックアップが必要な場合はValkeyやRedis OSSを採用する

### Amazon Athena

### AWS Database Migration Service(DMS)

### AWS DataSync

- データ転送の相手先がAWSサービスなら、IAM権限やセキュリティグループの許可が必要で、AWSサービス以外であればエージェントインストールが必要になる

  - DataSyncエージェントデプロイ
  - 転送ソース設定
  - 転送ターゲット設定
  - データ転送タスク設定、作成
  - データ転送タスク開始


### Amazon DocumentDB

## データ処理・分析サービス

### Amazon Simple Queue Service(SQS)
- メッセージをキューイングしてバッチのような非同期かつ並列の分散処理が可能になる


### Amazon Simple Email Service(SES)
- Eメール送信サービス

- SMS(ショートメッセージ)を送ることはできない


### Amazon Simple Notification Service(SNS)
- メッセージ通知サービス

- 任意のメッセージをHTTPSなどのさまざまなプロトコルでアプリケーションから簡単に送信できる（1:Nに通知ができる）

    - メッセージを送信し、通知を受信するためのアクセスポイントのこと
    - 作成後、購読者がこのトピックを後続することで通知が送受信できるようになる
    - 対象となるトピックから発信されるメッセージの購読者を設定できる
    - 作成したトピックに対して、アプリケーションアドからメッセージを発行する


### AWS Data Pipeline
- データベースからのデータ抽出・変換・保存(ETL: Extract Transform Load)など、データの順次処理が求められるケースがある

  - ETL処理をスケジュール機能で自動化
  - データの順次処理をワークフロー形式で定義
  - 処理の成功・失敗などのイベントの通知が可能
  - オンプレミス環境との連携が可能


### AWS Glue
- Data Pipelineと同様にデータの検出や加工、変換、結合などを行える


### AWS Step Functions
- データの順序処理に利用可能


### Amazon Kinesis
- リアルタイムに流れるデータの処理（ストリーミング処理）、IoTなどのデバイスからデータをリアルタイムに受信して分析する場合に利用

- SQSとの違いとして、複数のコンシューマが同じメッセージを同時に取得して処理できる

- **Kinesis Data Streams** ストリーミングデータの収集。EMRやLambdaなどに構築された独自アプリケーションと連携処理することができる
- **Kinesis Data Firehose** ストリーミングデータの保存。AWSの各サービスに配信、保存できるサービス
- **Kinesis Data Analytics** ストリーミングデータの分析。ストリーミングデータに対してSQLクエリを実行し、リアルタイム分析を行える


### Amazon Managed Service for Apache Flink
- ストリームデータに対し、ウィンドウ集計やフィルタリングなどの複雑なリアルタイム処理を実装できる

- Kinesis Data Streamから受け取ったストリームデータを処理して、Kinesis Firehoseに接続する


### Amazon EventBridge
- AWSの各種サービスや外部Saasなどのイベントソースで発生するイベントを、あらかじめ設定したルールに基づいて他サービスの各種ターゲットにリアルタイムで連携できるハブのようなサービス


### Amazon AppFlow
- SalesforceやSAP、SlackなどのSaaS製品と、AmazonS3やRedshiftなどのサービス間での安全なデータ転送を可能にする


### AWS Lake Formation
- データ処理・分析に必要なデータの蓄積場所であるデータレイクを簡易に構築・管理できる


### Amazon EMR
- Apache SparkやApache Hive、Prestoなどのオープンソースを使用した、ペタバイト・クラスのデータを処理、分析できるスケーラブルな処理サービス


### Apache Spark
- ストリーム処理(Spark Streaming)やグラフ分析(GraphX)のためのライブラリを備えた分散処理フレームワークであり、EMRクラスター上で実行するのに非常に適している


### Amazon DataZone
- AWS、オンプレミス、およびサードパーティのソース全体に保存されているデータを迅速かつ簡単にカタログ化、発見、共有、管理できるデータ管理サービス

- 組織のデータアセットを監督する管理者やデータスチュワードはきめ細かい制御を用いてデータへのアクセスを管理し、統制することができる

- 特定のパートナー企業とガバナンスを利かせながらセキュアに連携・共有する場合に最適

- AWS Data Exchangeはデータをサブスクリプション（販売、購入）形式で不特定多数または複数の企業に共有する場合に最適


### Amazon OpenSearch Service(Amazon Elasticsearch Service)
- Elasticsearchをベースとしたインタラクティブなログ分析・検索サービス


### AWS Data Exchange
- 世界中に存在する各種のサードパーティデータをAWSがサブスクリプション方式で提供する

## 構成管理サービス・開発サービス
  - 一般的にはシステム利用の需要を予測し、必要に応じて設備やサービスなどのリソースを提供できるように準備すること。クラウド環境においては複数のITリソースを、システムの利用状況や障害発生の状況に応じて動的に利用したり割り当てることを指す

  - 一般的にはアプリケーションを実行するためのファイルや、アセットを配布すること。クラウド環境においては、サーバにファイルやアセットを配置し、それが利用できる状況にすること


### AWS CloudFormation
- AWS内のすべてのインフラストラクチャリソースを自動でプロビジョニングできるサービス。テンプレートに記載された内容に基づいて、プロビジョニングしてスタックを作成するのがCloudFormationの機能

  - CloudFormationの設定ファイルを指し、プロビジョニングしたいAWSリソースを記述する

  - テンプレートに従ってCloudFormationでプロビジョニングされるAWSリソースの集合を指す


### AWS Control Tower
- AWSのマルチアカウントを管理するサービス


### AWS License Manager
- 各ベンダーの提供するライセンスが管理できるサービス


### AWS Service Catalog
- AWSでの使用が承認されたITサービスのカタログを作成・管理できるサービス


### AWS X-Ray
- 本番環境や分散アプリケーションを分析およびデバッグできるサービス


### AWS Proton
- コンテナとサーバーレス環境のためのアプリケーションデプロイサービス


## 運用管理サービス

### Amazon CloudWatch
- AWS上で稼働するさまざまなシステムやAWSのリソースの情報を収集、監視、可視化するサービス

- **メトリクス** 取得、監視する項目のこと
  - 標準メトリクス例: CPUUtilization, DiskReadOps, DiskReadBytes, NetworkIn
  - **カスタムメトリクス** 独自のメトリクスを定義して収集したい場合に使用する。メモリ使用率やディスク使用率、スワップ利用率の監視にはカスタムメトリクスが必要
  - 基本モニタリングは無料で、五分間隔で監視され、データは最大15ヶ月保持
  - 詳細モニタリングは追加料金が必要で、一分間隔で監視され、データは最大15ヶ月保持

- **アラーム** 監視している項目がある一定の値に達した場合にアラームとアクションを起動することができる。状態はOK, ALARM, INSUFFICIENT_DATA

- **CloudWatch Events / EventBridge** AWS上のリソースの状態を監視し、あるイベントをトリガーにアクションを実行する機能
  - 状態監視の対象はイベントソースと呼ばれ、スケジュールとAWSリソースに対するイベントの2種類がある
  - 監視後のアクションとして定義する対象をターゲットと呼ぶ、1つまたは複数設定できる

- **CloudWatch Logs** EC2インスタンスからログを収集する場合は、独自のエージェントをインストールする必要がある。CloudWatchのアラーム機能と連携してメールを通知するといったことが可能


### CloudTrail
- コンソールへのログインといった管理イベントとS3バケット上のデータ操作といったデータイベントの2種類を取得できる

- AWSアカウントを取得した時点で有効化され、過去90日間のサービスに対する操作を表示する

- 取得されたログはデフォルトでS3に保存され、CloudWatch Logsへの連携も可能

- AWSのさまざまなサービスをサポート・連携してログ記録を行う

- 記録内容は、サービスのAPIコール元、時間、送信元IPアドレス、呼び出したAPI、対象となるリソースなど


### VPCフローログ
- VPC内のネットワークインターフェイス間で行き来する通信の内容をキャプチャする機能

- 記録されるログとしては送信元IP・ポート、プロトコル番号など、ネットワーク通信に関する詳細な内容


### AWS Config
- AWSのサービスで管理されているリソースの構成変更を追跡するサービス


### AWS Systems Manager
- AWS内のさまざまなリソースの運用情報を統合的に可視化、制御するサービス


### AWS Trusted Advisor
- AWSのベストプラクティスに基づいて、コスト最適化、セキュリティ、耐障害性、パフォーマンス、サービスの制限の５項目で、ユーザーのAWS使用状況をチェックし、改善すべき事項を推奨するサービス

    - 特定のポートに対して無制限アクセスを許可しているセキュリティグループのルールをチェックする
  - ルートアカウントの多要素認証


### AWS Managed Grafana / AWS Managed Service for Prometheus
- オープンソースのGrafanaとPrometheusをAWSがマネージドサービスとして提供しているサービス群

- どちらも統合的な運用管理ツールとして、さまざまなメトリクスやログの収集、分析、可視化を行うことができる


### Amazon GuardDuty
- AWS内の各種ログを監視し悪意のある第三者による攻撃や不正操作などのセキュリティ脅威を検知するサービス


### AWS Inspector
- EC2インスタンスやAmazon ECRに登録されているコンテナイメージのセキュリティを高めるサービス


### AWS Artifact
- コンプライアンス関連の情報を一元管理するサービス

- AWSのセキュリティおよびコンプライアンスレポート(AWS Artifact Report)や秘密保持契約(NDA)などのAWSとの関連契約(AWS Artifact Agreements)をオンラインで管理・参照することができる


### AWS AuditManager
- AWSアカウント内のリソースがコンプライアンスに違反していないか、継続的にチェック・監査する仕組み

- 定期監査を仕組化し、インターネット・セキュリティ標準化フレームワークの1つであるCIS(Center for Internet Security)ベンチマークや、PCI DSS、GDPRなどの各種規格に沿って、継続的なチェックを自動化することができる


### Amazon Macie
- 機械学習とパターンマッチングを使用し、S3などに保存されている機密データを検出して保護する

- 個別識別情報(PII)などのデータを特定できる。


### AWS Security Hub
- AWS内のセキュリティの状態がセキュリティ標準およびベストプラクティスに準拠しているかどうかを包括的に把握し、アラートを一元化できる


## コスト管理

| タスク | タスク概要 | 主なAWSサービス |
| --- | --- | --- |
| コストの請求・支払い | 複数のAWSアカウントをまとめて支払い管理する | Consolidated Billing(一括請求) |
| コスト状況の詳細把握 | コストとリソース使用量の詳細をレポート分析する | AWS Cost and Usage Report(コストと使用状況レポート) |

| 過剰利用の監視 | 予算策定や予算超過のアラートを管理する | AWS Budgets(予算設定) |
| コスト最適化の検討 | コスト最適化の余地があるリソースを分析・解析する | AWS Trusted Advisor |

  - AWS Oragnizationsのと呼ばれるサービスの1機能として提供される


## 高可用性アーキテクチャ
- Design for Failure 障害発生を前提としたシステム構築

- 単一障害点 SPOF Single Point of Failure

- リソースの冗長化

- 地理的に離れた場所で冗長化

  - 一部分が故障したり動作不良を起こした場合に、その部分だけを切り離して復旧作業を行うことが可能になり、システム全体への影響を軽減することができたり、拡張しやすくなるというメリットもある


### ネットワークにおける高可用性の実現
  - 一方のAZが落ちてももう一方のAZでシステムを継続させることができる

  - AZ内では冗長化されているが、複数のAZ感では冗長化されていないため、複数のAZを横断するVPCネットワークを構成する場合、それぞれのAZでNATゲートウェイを配置するかどうかを検討する必要がある

- Direct Connect

- Route 53

  - 通常時はDirect Connectで障害発生時はSite-to-Site VPNへフェイルオーバーする
  - パフォーマンスに影響が出る可能性はある


- 「運用負荷をできるだけ最小限に抑えること」→マネージド型のものを使う

- RDSのI/O性能を向上させるための施作として、インスタンスタイプおよびストレージタイプの選定において、高いI/O性能を持つタイプを選ぶことが重要

- ランダムI/O遅延はDBの性能不足が原因

  - AutoScalingグループでRDSのDBインスタンスの起動時間をスケジュールするのも無理

  - メモリを逼迫する時間帯が分かっているからといって必ずその時間帯にスケールするようにするのではなく、一応様子見るようにヘルスチェック入れるのが良いように思われ

  - オンデマンドインスタンスのキャパシティ予約

  - EC2インスタンスにインストールされるソフトウェアコンポーネント

  - EC2タイプではEC2のステータスがrunning以外である場合やシステムステータスがimpairedである場合にそのインスタンスを異常とみなす
  - ELBタイプではインスタンスのステータスチェックとELBのヘルスチェックを基に、インスタンスの状態を評価する

  - レプリケーションを行うためにはオブジェクトのバージョ二ングが有効である必要があり、次にソースバケットの管理設定が目にアクセスし、レプリケーションルールを追加して宛先バケットを指定する

- 古いEC2インスタンスからEBSボリュームを切り離すには古いEC2インスタンスを停止した上で、ボリュームをデタッチ

- AuroraにはマルチAZ配置を有効化するという操作はない、RDSにはある

- 静的WEBホスティングで403になる場合はバケットポリシーでバケットの読み取り許可設定する（ブロックパブリックアクセスを無効化した後）

    - ルートテーブルにエントリを追加

    - VPC内にENI(Elastic Network Interface)

- Route53 シャッフルシャーディング

- リザーブドインスタンス（スタンダード）はインスタンスファミリーやOSを変更できないけど、コンバーティブルは変更可能
