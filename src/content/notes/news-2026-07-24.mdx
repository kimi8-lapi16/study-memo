## OpenAI、企業向けAIエージェント基盤「Presence」を発表

OpenAIは7月22日、企業が音声・チャットのAIエージェントを構築・運用できる新プラットフォーム「Presence」を発表した。

AI・LLM動向案件。モデルへの生API提供にとどまらず、企業の業務ポリシーやガードレール、テスト、エスカレーションの仕組みまで組み合わせて「本番運用に耐えるエージェント」を管理できるマネージドサービスへ軸足を移した点が特徴。現時点ではセルフサービス提供ではなく、OpenAIのフォワード・デプロイド・エンジニアや一部SIが導入を主導する限定提供。OpenAI自身の英語圏電話サポート窓口にも導入されており、導入から数週間で人手対応と同等以上の品質基準を満たし、着信の約75%を自律的に解決しているという。カスタマーサポートのほか銀行・通信・保険業界での早期検証も進んでいる。

## Google対SerpApi訴訟、「検索結果は著作物ではない」とGoogleが敗訴

検索結果をスクレイピングして再販するSerpApiをGoogleがDMCA違反で提訴していた裁判で、米カリフォルニア北部地区連邦地裁が7月23日までにGoogle敗訴の判断を示した。

訴訟・トラブル案件。イヴォン・ゴンザレス・ロジャース判事は、URLやスニペット、インデックスデータといった集約された検索結果は一般に公開された事実であり著作権法で保護される著作物には当たらないと判断。DMCA第1201条に基づくアクセス制御回避の主張についても、Googleが原著作権者から保護のための認可を得ていなかったとして退けた（Googleには21日以内の訴状修正・再提出の機会が与えられている）。SerpApiのCEOは「オープンなインターネットに支えられているすべての人の勝利」とコメントしている。

出典: [GIGAZINE](https://gigazine.net/news/20260723-google-loses-lawsuit-against-serpapi/)

## Check Point SmartConsoleに認証バイパスの重大脆弱性、悪用を確認

Check Pointは、Security Management ServerのSmartConsoleログイン処理に関する認証バイパスの脆弱性（CVE-2026-16232、CVSS 9.1）が実際に悪用されていると7月23日までに公表した。

セキュリティインシデント・脆弱性案件。未認証の攻撃者がアプリケーショントークンを取得することで管理者権限を伴う認証を偽装でき、管理サーバーへの侵入後はセキュリティ設定・ポリシーを改変できてしまう。悪用が確認されているのは管理インターフェースをIP制限なしにインターネットへ露出させた一部の構成に限られるという。CISAは同脆弱性を既知の悪用済み脆弱性（KEV）カタログに追加し、米連邦機関に7月25日までの対応を指示した。

<Note label="⚠ 注意">
パッチ適用に加え、管理アクセスを信頼できるIPアドレス・サブネットに制限し、SmartConsoleのログで「Authentication method: application token」を検索して侵害の有無を確認することが推奨されている。
</Note>

## Linuxカーネルの10年物レースコンディション「RefluXFS」、root権限奪取が可能に

Qualysは7月22日、XFSファイルシステムのreflink（コピーオンライト）処理に起因するLinuxカーネルのレースコンディション脆弱性「RefluXFS」（CVE-2026-64600）を公表した。

セキュリティインシデント・脆弱性案件。共有ブロックへの書き込み中に発生する競合状態を突くことで、通常権限のローカルユーザーが保護されたファイルを上書きし、SELinuxをEnforcingモードで運用している環境も含めてroot権限を奪取できる。原因となるコードは2017年公開のLinux 4.11から存在しており、XFSを標準ルートファイルシステムとするRHELやOracle Linux、Amazon Linux、Fedora Serverなど、世界で1,640万台超のシステムに影響する可能性があるとQualysは推定している。修正は7月16日にLinux本流へマージ済みで、各ディストリビューションがバックポートを進めている。

出典: [Qualys](https://blog.qualys.com/vulnerabilities-threat-research/2026/07/22/refluxfs-a-linux-kernel-local-privilege-escalation-to-root-in-xfs-cve-2026-64600), [BleepingComputer](https://www.bleepingcomputer.com/news/linux/new-refluxfs-linux-flaw-lets-attackers-gain-root-privileges/)

## 今日のTips: Prismaの対話型トランザクションはデフォルト5秒でタイムアウトする

Prismaの`$transaction`をコールバック形式（インタラクティブトランザクション）で使う場合、トランザクション内の処理がデフォルトの`timeout`（5000ms）を超えると自動的にロールバックされてエラーになる。トランザクション取得待ちの上限を示す`maxWait`（デフォルト2000ms）とは別の設定値なので混同しないよう注意したい。

外部APIへのリクエストのような時間のかかる処理をうっかりトランザクション内に含めてしまうと、本番負荷時にこのタイムアウトへ引っかかりやすい。必要なら第2引数でオプションを指定して調整できる。

```ts
await prisma.$transaction(
  async (tx) => {
    await tx.order.create({ data: orderData });
    await tx.inventory.update({ where: { id }, data: { qty: { decrement: 1 } } });
  },
  { maxWait: 2000, timeout: 5000 } // ミリ秒単位
);
```

<Tip label="▶ 補足">
公式ドキュメントも「トランザクションは短く保つこと。長時間実行されるトランザクションはパフォーマンス低下やデッドロックの原因になる」と明記しており、外部I/Oはトランザクションの外に出すのが基本方針となる。
</Tip>
