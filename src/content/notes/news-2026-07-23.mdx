## SharePointの新たな未認証RCE脆弱性、公開PoC直後から悪用が拡大

Microsoft SharePoint（オンプレミス版）の未認証リモートコード実行の脆弱性について、実証コード公開直後から実際の悪用が急拡大していることが分かった。

セキュリティインシデント・脆弱性案件。脆弱性はCVE-2026-50522（深刻度Critical）で、逆シリアル化の不備により認証なしにリモートコード実行が可能。7月20日に実証コードが公開されると数時間以内に攻撃者が悪用を開始し、セキュリティ企業watchTowrのハニーポットでも侵害の成功例が検出された。攻撃者は侵入後にSharePointの「マシンキー」を窃取しており、これを使えば正規ユーザーになりすます認証トークンを偽造できるため、パッチを適用しただけでは不十分で、露出していた資産の認証情報のローテーションが別途必要になる。

7月のセキュリティ更新プログラムで修正済みだが、悪用がパッチ適用前提を崩す形で進んでいるため、パッチ適用に加えて資格情報の再発行が推奨される。
</Note>

## 今日のTips: NestJSのグローバルパイプは登録方法次第でDIが効かなくなる

`app.useGlobalPipes()` でグローバルパイプを登録すると、そのバインディングはどのモジュールの文脈にも属さない状態で行われるため、パイプの中でサービスなどの依存関係をDIで注入できない。

DIを効かせたい場合は、`APP_PIPE` トークンを使ってモジュールの `providers` としてグローバルパイプを登録すればよい。この方法ならどのモジュールで定義してもグローバル扱いになり、通常のプロバイダと同様にコンストラクタインジェクションが使える。

```ts
import { Module } from '@nestjs/common';
import { APP_PIPE } from '@nestjs/core';

@Module({
  providers: [
    { provide: APP_PIPE, useClass: ValidationPipe },
  ],
})
export class AppModule {}
```

<Tip label="▶ 補足">
ハイブリッドアプリでは <Mono>useGlobalPipes()</Mono> はゲートウェイやマイクロサービスにパイプを適用しない、という制約もあわせて覚えておきたい。
</Tip>
