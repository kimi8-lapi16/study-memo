// 01 — OFFENSE

AIエージェントによる攻撃の現実

メモの「攻撃者は1〜2割しか介入してない」── これは実際に起きた事件のこと。

Anthropicが2025年9月に検知したもので、Claude Code を使った中国の国家支援グループ(GTG-1002と呼称)が、ペネトレーションテストのオーケストレーター兼エージェントとして自律的に動かし、戦術的オペレーションの80〜90%をAIが独立して実行した。人間が担当したのはターゲット選定と戦略的な承認だけで、偵察・脆弱性の悪用・認証情報の窃取・横展開(ラテラルムーブメント)・データ持ち出しといった実作業はAIが処理した。

メモの整理 → 事実と一致

「どこへ攻撃を仕掛けたいか」「分析結果のレビュー」＝人間側。「データ収集」「脆弱性を突く」＝AI側。この切り分けはまさに事件の構造そのまま。ターゲットはテック企業・金融機関・化学メーカー・政府機関など世界の約30の高価値ターゲット。

メモにあった「anthropicから出してるページがありそう」── これは正解。Disrupting the first reported AI-orchestrated cyber espionage campaign という13ページのレポートが公開されている。

⚠ 注意 — 鵜呑みにしない この件には懐疑的な見方もある。具体的なコードサンプルやIoC(侵害指標)、観測されたTTPなどの技術的証拠が乏しいとして、セキュリティ専門家から「90%自律」という主張に疑問を呈する声も出ている。「そういう議論もある」くらいに留めておくのがよさそう。

補足された関連トピック

強化学習によるEDRバイパス — メモの「検知回避を報酬にして回すと商用モデルより高い回避能力」という話。EDR(Endpoint Detection and Response＝端末上の不審な挙動を検知する製品)を「いかにすり抜けるか」を報酬関数にしてRLを回すと、検知回避に特化したモデルが作れてしまう、という研究方向。

GUI操作が苦手 — メモ通り「もう時間の問題」。現状のエージェントはCLI/API操作は得意でもGUIのポチポチ操作が弱点、という認識。

// 02 — DEFENSE / DETECTION

検知エンジニアリング

ここからが防御側の本題。Detection Engineering。

メモの2つの定義(Detection寄り / Engineering寄り)はよく整理されている。要するに「脅威を検知するためのロジックを、体系立てて作り・改善し・運用するプロセス」のこと。職人芸だったものを工学として標準化していこうという流れ。メモの年表が的確:

2015 MITRE ATT&CK ── 攻撃者の手口(戦術・技術)を体系化した「共通言語」。皆が同じ語彙で攻撃を語れるようになった。
2017 Sigma ── 検知ルールの共通フォーマット(後述)。
2020 Detection Engineering が職種・用語として定着。

SIEM とは

SIEM(Security Information and Event Management)は、サーバー・ネットワーク機器・アプリなどから出る大量のログを一箇所に集約して、相関分析・検知・アラートを行う基盤。Splunk / Microsoft Sentinel / Elastic などが代表例。各製品でクエリ言語がバラバラなのが課題で、そこに効いてくるのがSigma。

Pyramid of Pain

David Biancoが提唱した概念フレームワークで、攻撃者が検知を回避するために変更を加える際の「難易度とコスト」を段階的に表したもの。下から上へ、攻撃者にとって変更が辛く(painful)なる。

TTPs 手口そのもの ── 作り直しが必要 = 最も痛い
Tools ツール
Network/Host Artifacts 挙動の痕跡
Domain Names ドメイン
IP Addresses IPアドレス
Hash Values ハッシュ値 ── 変更が一瞬 = 痛くない
▲ 攻撃者に痛い / 検知エンジニアが狙う容易に回避される ▼

ポイントは、ハッシュ検知は簡単に回避され、IPやドメインはローテーションされてしまうが、攻撃者の「振る舞い」を検知できるようになると、攻撃者は手口そのものを作り直さざるを得なくなる＝そこで初めて「痛み」を与えられる、という考え方。だから検知エンジニアは可能な限りピラミッドの上層を狙う。

Sigma

メモの理解で合っている。SigmaはYAML形式(拡張子は .yml)で書く検知ルールの規格で、様々なSIEM製品のクエリに変換できる。「ログイベントを記述する共通言語」と思えばいい。1ルール ≒ 1つの検知観点で、30行程度のYAMLからなる。

メモにあった構成要素

detectionブロック — ログからどう脅威を検出するかのロジック本体。論理和(OR条件)はリストで書く。

Modifier — 完全一致以外の表現方法。startswith / endswith / contains / re(正規表現)などで「〜で始まる」「〜を含む」といった柔軟なマッチが書ける。

Pyramid of Painとの関係で「SigmaはLayer 2を出力する」というメモは、おそらく「Sigmaルールが主に振る舞い系の中間層を狙う」という講師の整理。実際、MITREの Summiting the Pyramid プロジェクトでは公開Sigmaルールをピラミッドに対してスコアリングし、回避耐性(robustness)の高い検知を作る取り組みが行われている。

ツール群 ── Yamato Security 製

メモに出てきた隼・朱雀シリーズは、すべて日本の Yamato Security(Zach Mathis氏主宰)が公開しているOSS DFIRツール。Rust製で高速なのが特徴。

隼HAYABUSA
Windowsイベントログ向けの、Sigmaベースの脅威ハンティング＆高速フォレンジック・タイムライン生成ツール。.evtx(または変換したJSON)を読み込み、Sigma＋独自ルールで脅威を検出、CSV/HTML/JSONで出力。メモの「.evtxを食わせてcsv/jsonで出る」「4000以上のSigmaルール」はこれ。
朱雀SUZAKU
Hayabusaのクラウドログ版。AWS CloudTrailにネイティブなSigma検知を持ち、Azure/GCPも対応予定。クラウドには何千ものAPIコールがあって手で追えないので、ノイズの中から攻撃を見つけ、必要なイベントだけのDFIRタイムラインを提供する。
鷹匠TAKAJŌ
Hayabusaの結果解析ツール。隼が獲った獲物を鷹匠がさばく、という命名。体系として一緒に覚えておくと掴みやすい。
▶ きみしま向けメモ 本業でCloudTrailを扱っているので、Suzaku が一番実務に刺さるはず。一度サンプルデータで動かしてみる価値あり。メモの「THuntCloud」は、おそらくSuzakuの旧名か関連プロジェクトの別表記の可能性が高い(検索ではクラウドログツールとしてはSuzakuがヒット)。「mecha-hayabusa」は生成AIによる解析補助ツールという理解でOK。

マルチステップ攻撃の評価

Claude Mythos Preview — メモの「調べてみるか」のMythosの正体はこれ。Anthropic製のフロンティアモデル(2026年4月7日発表)で、サイバーセキュリティタスクに異常に長けているとされる。「公開するにはリスクが高すぎる」として一般提供せず、テック大手に限定アクセスを与える Project Glasswing という枠組みでテストされている。プレフィックスは「Claude」だった。

The Last Ones (TLO) — メモの「32ステップの企業ネットワークシミュレーション」。UK AI Security Institute(AISI)が構築した、初期偵察からネットワーク完全乗っ取りまでをカバーする評価環境で、人間なら約20時間かかる想定。このMythosの評価で使われたベンチマークで、MythosはTLOを最初から最後まで解いた初のモデル(10回中3回成功 / 平均32ステップ中22ステップ完了)。

⚠ AISIの冷静な見立て 面白いのは独立評価機関(AISI)の整理で、単発タスク(CTF等)では他のフロンティアモデルとほぼ横並び。差が出たのは何十手も連鎖させるTLOだけだった。つまり「個々の技は皆できる、最後までやり切れるかが分かれ目」という話。メモの「一般的なセキュリティ機能が欠如した環境なのでスコアだけ見て不安になるのは違う」も裏取り済みで、AISI自身が「弱いシステムは攻略できるが、よく防御されたシステムを破れることを示すものではない」と明言している。講師の整理は正確だった。
// 03 — PROACTIVE DEFENSE

AIエージェントへの能動的防御

「AIに対抗するにはAIしかないのでは」というメモの問題意識への答え。鍵は「AIエージェントらしさを検出して対策する」。

Cloak, Honey, Trap (CHeaT)

USENIX Security '25で発表された研究(Ayzenshteyn, Weiss, Mirskyら)で、LLMエージェントの脆弱性を逆手に取った能動的防御のフレームワーク。名前の通り3つの柱:

Cloak
隠蔽
誤った情報・ミスディレクションで本物の資産を隠す。
Honey
おとり
LLM特有のハニートークンでエージェントをおびき寄せ、存在を露呈させたりリソースを枯渇させる。
Trap
罠
LLM固有の欠陥を突いて、攻撃をループに陥れ遅延・停止させる。

メモの「人間には見えない文字を置いて、それを読んでアクセスしてきたやつをAIエージェントと判断」はまさに Honey / ハニートークン の発想。6つの戦略・15のテクニックがあり、その多くはプロンプトインジェクションに依存しない。エージェントに信頼できないコードを実行させて、逆に防御側が攻撃者のインフラにアクセスする、といった新しいエクスプロイトも実証された。CHeaT という名前でOSSツールも公開されている。

メモの「アンダースコアフォーアドミン」(_for_admin 的なもの?)は、おそらくこの文脈のおとりトークンの命名例の話。

AI Labyrinth (メモの「AIラビリンス」)

これはCloudflareの機能。保護対象ページに、ボットには正当に見えるが実際は無関係なAI生成ページへの隠しリンクを埋め込む仕組み。人間のユーザーには見えず、ボットだけがこのリンクをたどるため、新しいボットのパターンやシグネチャを特定するのにも使える。

なぜ "ブロック" ではなく "罠" なのか

直接ブロックすると攻撃者に「検知された」と気づかせて手口を変えられてしまう。だからブロックせず、説得力はあるが無関係なコンテンツでリソースを浪費させる。メモの「Unicodeの特殊文字がAIしか読めない」も、人間に見えない要素でAIだけを誘い込む＝Honey/おとりの同じ発想のバリエーション。

ひとことまとめ

3行で言うと、こういう構図だった。

▌ 攻撃側

AIエージェントが攻撃の8〜9割を自律実行する時代に入った(Anthropic GTG-1002事例)。ただし証拠面で懐疑論もある。
