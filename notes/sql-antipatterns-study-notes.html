// 01 — FUNDAMENTALS

アンチパターンについて

すべての問題は命名とわかりづらい設計から始まる。

データベース設計における「アンチパターン」とは、一見解決策に見えるが、実際には後で問題を引き起こす設計パターンのこと。特にRDBMSでは、初期の設計判断が後々まで影響を及ぼすため、よくある失敗パターンを知っておくことが重要。

なぜアンチパターンを学ぶのか

命名の曖昧さ、正規化の過不足、制約の不適切さ──これらは初期段階では気づきにくいが、システムが成長するにつれてパフォーマンス問題、保守性の低下、データ整合性の崩壊として表面化する。アンチパターンを知ることで、設計時に「これは後で問題になる」と気づけるようになる。

// 02 — HISTORY

失われた事実

履歴を保存していないことで、必要な計算や分析ができなくなる問題。

多くのシステムでは「現在の状態」だけを保存し、「どう変化したか」を記録していない。しかし、ビジネス要件が変わり「過去の推移を知りたい」となったとき、履歴がないと遡って計算することができない。

よくあるケース

  • ユーザーのステータス変更履歴がなく、「いつプレミアム会員になったか」が分からない
  • 在庫数の更新履歴がなく、「なぜこの数になったか」が追跡できない
  • 価格変更の履歴がなく、「過去のある時点での売上」が再計算できない

解決策とトレードオフ

基本的には履歴テーブルを作るだけで解決する。しかし、すべてのテーブルに履歴を持たせるとストレージコストとクエリの複雑さが増す。

⚠ 設計時の判断基準 「履歴がないと後で困るか？」を設計段階で考慮する。全部履歴化するのではなく、ビジネスロジック上「過去の状態を知る必要があるか」を見極める。監査要件、分析要件、コンプライアンスなどの観点で判断すること。
// 03 — JOIN

やりすぎたJOIN

JOINはRDBMSにとって負荷の高い処理。安易な多段JOINはパフォーマンス悪化の元凶になる。

JOINが重い理由

JOINは複数のテーブルを組み合わせる際に、内部でネステッドループ、ハッシュ結合、ソートマージといった処理を行う。テーブルサイズが大きくなるほど、また結合するテーブル数が増えるほど、計算量は指数関数的に増加する。

対処法

  • JOINは必要最小限に — 本当に必要なテーブルだけを結合する
  • INDEXを有効活用 — JOIN条件となるカラムに適切なINDEXを貼る
  • JOINするテーブルを小さくする — WHERE句でまず絞り込んでからJOINする
  • 複雑なクエリはVIEWにする — 保守性向上と、場合によってはマテリアライズドビューで高速化
設計時の工夫

頻繁にJOINされるデータは、あえて非正規化して埋め込む選択肢もある。正規化は美しいが、パフォーマンスとのトレードオフを常に意識すること。特に参照頻度が高く更新頻度が低いマスタデータなどは、非正規化の恩恵が大きい。

// 04 — INDEX

効かないINDEX

INDEXを貼ったのにクエリが遅い──よくある「INDEXが効かない」パターンを知る。

B-Tree INDEX の基本

RDBMSで最も一般的なINDEXはB-Tree構造。検索・挿入・削除がすべて O(log n) で処理できる優れた構造だが、すべてのケースで効くわけではない。

INDEXが使われない典型的なケース

  • 検索結果が多すぎる — 全体の20〜30%以上ヒットする場合、フルスキャンの方が速いと判断されることがある
  • 条件にその列を使っていない — WHERE句に登場しない列のINDEXは使われない
  • カーディナリティが低い — true/falseのような値の種類が少ない列ではINDEXの効果が薄い
  • あいまい検索 — LIKE '%keyword%' のような前方一致でない検索はINDEXが効かない
  • 統計情報と実データの乖離 — ANALYZE/VACUUM不足でオプティマイザが誤判断する
▶ 実務での確認 EXPLAIN / EXPLAIN ANALYZE を使って実行計画を確認する習慣をつける。「INDEXを貼ったから速い」と思い込まず、実際に使われているかを必ず検証すること。PostgreSQLなら pg_stat_statements、MySQLなら slow_query_log でボトルネックを特定できる。
// 05 — FLAGS

フラグの闇

ステータスフラグの安易な使用は、クエリの複雑化、制約の喪失、パフォーマンス低下を招く。

フラグがもたらす問題

  • クエリが複雑になる — WHERE句に複数のフラグ条件が並び、可読性が著しく低下
  • UNIQUE制約が使えない — 「削除されていないレコードの中で一意」といった制約が表現できない
  • カーディナリティが低くなる — true/false の2値ではINDEXが効きにくい
  • フラグの組み合わせ爆発 — is_deleted, is_active, is_verified... と増えるほど、状態の組み合わせが把握できなくなる

解決策

根本的な解決策は、異なる状態のレコードを1つのテーブルに混在させないこと。つまりテーブルごと分けるか、VIEWで分ける。

deleted_at も実質フラグである

「is_deleted の代わりに deleted_at を使う」という対処もよく見るが、これは単なる言い換えに過ぎない。WHERE deleted_at IS NULL と WHERE is_deleted = false は本質的に同じで、削除済みと未削除が同じテーブルに混在する問題は解決していない。カーディナリティ、UNIQUE制約、クエリ複雑化といった問題もそのまま残る。

正しいアプローチ

  • 物理削除 + 履歴テーブル — 削除されたレコードは deleted_users テーブルに移動。メインテーブルは常に「有効なレコードだけ」が入っている状態を保つ
  • VIEWで論理分離 — 物理的には1テーブルでも、active_users VIEW / deleted_users VIEW を作り、アプリケーションからは常にVIEW経由でアクセス。元テーブルへの直接アクセスは禁止
  • そもそもフラグを持たない設計 — 「削除」ではなく「契約終了日 (contract_end_date)」のような業務的事実を記録し、WHERE contract_end_date IS NULL OR contract_end_date > CURRENT_DATE で有効なレコードを取得
users テーブル (全レコード混在) ──▶ users + deleted_users に分離 or VIEWで論理分離

ポイントは、「状態」ではなく「事実」を保存すること。is_deleted や deleted_at は「削除された状態」を表すフラグだが、contract_end_date は「契約終了という事実」を表す。後者の方がビジネス的意味が明確で、クエリもシンプルになる。

// 06 — ORDER BY

ソートの依存

RDBMSの根底は集合論。順序の概念はないため、ソートは実は苦手な領域。

SQLの評価順序を理解する

SQLは書かれた順番とは異なる順序で評価される。この理解がないと、なぜINDEXが効かないのか、なぜソートが遅いのかが見えてこない。

エグゼキュータによる評価順序
1. FROM — テーブル指定 2. ON — 結合条件 3. JOIN — テーブル結合 4. WHERE — 絞り込み 5. GROUP BY — 集約 6. HAVING — 集約後フィルタ 7. SELECT — 列選択 8. DISTINCT — 重複排除 9. ORDER BY — ソート 10. LIMIT — 件数制限

ソートを速くする戦略

WHERE句でまず絞り込む — ORDER BYの前にWHERE句で評価されるため、対象レコードを減らせばソートコストが下がる。WHERE句に対応するINDEXを貼ることが第一優先。

ORDER BY句狙いのINDEX — B-Tree INDEXは内部でソート済みの状態で格納されているため、ORDER BYで指定するカラムにINDEXがあれば、ソート処理そのものがスキップされる。

⚠ ページング最適化 OFFSET 10000 LIMIT 10 のような深いページングは、内部で10000件スキップする処理が走るため遅い。代わりに WHERE id > {前ページ最後のid} ORDER BY id LIMIT 10 のようなカーソルベースページングを使うと劇的に速くなる。
▶ ある日突然遅くなる データ量が増えたり統計情報が古くなると、ある日突然同じクエリが遅くなることがある。定期的に実行計画を確認し、想定通りのINDEXが使われているか監視すること。
// 07 — EAV

隠された状態

カラム名に用途を詰め込む設計は、アプリケーション側に複雑なロジックを押し付ける。

EAV (Entity Attribute Value) の罠

EAVとは、エンティティ・属性・値の3つのカラムで任意の属性を表現する設計パターン。一見柔軟に見えるが、RDBMSの強みをすべて捨てることになる。

EAVの問題点
  • Value列の型が定まらない（すべてTEXT型にせざるを得ない）
  • CHECK制約、NOT NULL制約、外部キー制約が使えない
  • 集計・検索のたびに複雑なJOINやCASE文が必要になる
  • パフォーマンスが著しく悪化する

同様に、status_1, status_2, flag_type_a のようなカラム名も避けるべき。カラム名以上に用途を含めると、「このカラムが何を意味するか」をアプリケーションコードでハードコードすることになり、データベースが自己文書化されない。

正しい設計

属性が可変なら、素直に列を追加するか、本当に必要ならJSON型（後述の制約あり）を使う。属性が固定されているなら、専用のカラムとして定義し、適切な型と制約を設定する。

// 08 — JSON

JSON型の甘い罠

JSON型は便利だが、安易に使うとスキーマレスの混沌を招く。

よくある失敗例

Web APIのレスポンスをそのまま詰める — バージョン変更でAPIのレスポンス構造が変わると、同じカラム内に異なる構造のJSONが混在する。過去のレコードがパースできなくなったり、検索条件が書けなくなる。

JSON型を使わない方がいい判断基準

  • 正規化できるなら使わない — 構造化できるデータは素直にテーブルとカラムに分割する
  • 頻繁に更新するなら使わない — JSON内の一部だけ更新する操作は非効率で、競合も起きやすい
  • 検索条件として固定できるなら使わない — JSON内の特定キーで頻繁に検索するなら、それは独立したカラムにすべき
⚠ JSON型が適切なケース 完全に任意のメタデータ、ログのペイロード、監査証跡など、構造が予測不可能で検索・更新の頻度が低いものならJSON型が適している。ただしその場合でも、スキーマバージョンを含めるなど、将来の自分を助ける工夫をすること。
// 09 — CONSTRAINTS

強すぎる制約

早すぎる最適化、強すぎる制約は、時として開発者に牙を剥く。

制約が裏目に出る例

  • 独自のカラム型定義 — 厳密すぎるカスタムドメイン型により、テストデータのINSERTができない
  • updated_at >= now() のCHECK制約 — 過去のバックアップをリストアするとき、未来の日付扱いになり制約違反で失敗する
  • ビジネスルールの制約化 — 「未成年は購入不可」のようなルールをCHECK制約にすると、法改正で地獄を見る

制約の4段階

レベル説明例
制約なし型の値域のみTEXT型に何でも入る
弱い制約データ構造を守る最低限NOT NULL, UNIQUE, 外部キー
強い制約RDBMS機能で適切に防御CHECK制約, EXCLUDE制約
強すぎる制約ビジネスルール・仕様レベル年齢制限, 営業時間制約
適切なバランス

「強い制約」までは推奨。データ整合性を守るための構造的制約は積極的に使うべき。しかし、ビジネスルールや運用都合の制約はアプリケーション層で担保し、データベースには持ち込まない。特にリストア・マイグレーション時の柔軟性を損なう制約は避けること。

// 10 — BACKUP & RECOVERY

転んだ後のバックアップ

バックアップは取るだけでは意味がない。リストアできて初めて価値がある。

バックアップの3つの手法

論理バックアップ

SQLやCSV形式でデータをエクスポート。mysqldump, pg_dump などのコマンドで取得。可搬性が高く、別バージョンのDBにもリストアできる。ただしサイズが大きいとダンプ・リストアに時間がかかる。

物理バックアップ

データベースの物理ファイル（データディレクトリ）をまるごとコピー。cp, rsync や専用ツールを使用。高速だが、同じバージョン・同じアーキテクチャでないとリストアできない。停止が必要なケースもある。

PITR (Point-In-Time Recovery)

特定の日時の状態にデータをリストアできる手法。ベースバックアップ + トランザクションログ（MySQLならバイナリログ、PostgreSQLならWALアーカイブ）で実現。秒単位での復旧が可能だが、運用が複雑。

バックアップ設計の3つの指標

  • RPO (Recovery Point Objective) — どの時点まで遡って復旧するか（データロスの許容範囲）
  • RTO (Recovery Time Objective) — どれくらいの時間で復旧できるか（ダウンタイムの許容範囲）
  • RLO (Recovery Level Objective) — どこまで復旧するか（復旧レベル：全体か一部か）

稼働率とバックアップ設計

稼働率年間停止時間求められる対策
90%36.5日バックアップとリストアで十分
99%3.65日予備マシン、リストア所要時間の把握
99.9%8.7時間24/365サポート、法定停電対応
99.99%52分コールドスタンバイ必須
99.999%5分遅延レプリケーション等の専用システム
99.9999%32秒無停止サーバ、コスト極大
⚠ 運用の鉄則
  • バックアップが正しく取れているか毎回確認 — 失敗を見逃さない仕組みを作る
  • 定期的にリストアを実行 — 「バックアップがあるから安心」は幻想。実際に戻せて初めて安心
  • 手順書を整備 — 緊急時にパニックしないよう、具体的なユースケース別の手順を用意
// 11 — MONITORING

見られないエラーログ

エラーは出ているのに、誰も気づかない。通知が届かない。それでは監視していないのと同じ。

監視の本質

ログを吐いているだけでは意味がない。重要な通知を、知るべきすべての人に、すばやく・正しく・漏れなく届けることが監視の目的。

  • 誰に届けるか — オンコール、開発チーム、SRE、それぞれ必要な情報が違う
  • どう届けるか — Slack、PagerDuty、メール、SMSなど、緊急度に応じた手段
  • 何を届けるか — ノイズが多すぎるとアラート疲れで無視される。重要度の判別が肝
▶ ログの棚卸し 定期的に「どんなログが出ているか」を確認する習慣をつける。想定外のエラーが毎日吐かれているのに誰も気づいていない、というのはよくある話。ログの種類・頻度・内容を定期レビューし、重要なものには必ずアラートを設定すること。
アラート設計の原則
  • アクション可能なアラートだけを送る（「知ってどうする？」がないものは送らない）
  • 誤検知を最小化する（オオカミ少年になると誰も見なくなる）
  • アラートには必ずコンテキストを含める（「エラーが出た」だけでなく「どこで何が」まで）
