# メモ
## 1章 アンチパターンについて解説
- 命名やわかりづらい設計がすべて

## 2章 失われた事実
- 履歴を保存してないことにより、必要な計算ができない
- 解決策としては、履歴を保存するだけだが、トレードオフである
  - 毎回常に全部履歴を取る必要があるわけじゃない。履歴として持たないと不都合が生じるかを設計段階の考慮に入れる

## 3章 やりすぎたJOIN
- JOINはDBにとってはかなり負荷の高い処理であることを理解しておく必要がある
- JOINは必要最低限
- INDEXを有効活用する
- JOINするテーブルは小さくしてJOINする
- 複雑なクエリになったらVIEWにする

## 4章 効かないINDEX
- INDEXによく使われるのはB-Tree
- INDEXが使われないケースとしては
  - 検索結果が多い、全体の件数が少ない
  - 条件にその列を使っていない
  - カーディナリティの低い列に対する検索
  - あいまいな検索
  - 統計情報と実際のテーブルで乖離がある場合

## 5章 フラグの闇
- クエリが複雑になる
- UNIQUE制約が使えない
- カーディナリティが低くなる
- 事実のみを保存するために、テーブルごと分ける or VIEWで分ける

## 6章 ソートの依存
- そもそもRDBは集合論が元になっているので、順序はRDBMSが実現してくれてるものになるので、ソートは苦手な領域
- SQLの評価順序
  - クエリの受信
  - 構文解析
  - 書き換え
  - 実行計画生成/最適化
  - 実行(エグゼキューたによる評価順序を更に参照)
  - 実行結果送信
- エグゼキュータによる評価順序
  - FROM
  - ON
  - JOIN
  - WHERE
  - GROUP BY
  - HAVING
  - SELECT
  - DISTINCT
  - ORDER BY
  - LIMIT
- WHERE句狙いのINDEX
  - ORDER BYの前にWHERE句の評価があるので、そこで十分に絞り込んでおけば、そこまでソートするのが大変じゃないこともある
- ORDER BY句狙いのINDEX
  - B-Tree INDEXがソート済みの状態で格納されているので、そこから取れるとソートが不要になる
- ページングで遅くなる場合にはWHERE句にidとして前ページの最後のものを渡すのもある
- ある日突然クエリが遅くなる可能性があるので、実行計画を見る
  
## 7章 隠された状態
- カラム名以上に用途を含めるとアプリ側に複雑なロジックを書く必要が出てくる
- EAV(Entitiy Attribute Value) キーとバリューで二つのカラム、特にバリューの型を適切につけられなかったり、制約をつけられなかったりとDBの強みを自ら捨てることになる

## 8章 JSON型の甘い罠
- WebAPIの戻り値を詰めてしまうと、バージョン変更などで同じカラムでもカラムが違うことになる
- JSON型を使わないようにする問いかけ
  - 正規化できるなら使わない
  - 頻繁に更新をしたいなら使わない
  - 検索条件としてJSON内の要素が固定できるなら使わない

## 9章 強すぎる制約
- 独自のカラム型定義により、自前でのINSERTできなかったり
- updated_at >= now() としてしまうことで、restoreに失敗してしまったり
- 上記のよう早すぎる最適化、強すぎる制約は時として牙を向くことがある
- 制約としては下記のように4つあると考えられている
  - 制約なし
    - 制約が存在せず、型自体が規定する値域であれば、なんでも入れられる状態
  - 弱い制約
    - NOT NULL制約、UNIQUE制約、外部キー制約、データ構造を守るうえで必要最低限の制約がかかった状態
  - 強い制約
    - CHECK制約やEXCLUDE制約など、RDBMSの機能に依存するが、適切に使うことでデータを的確に守ることができる制約がかかった状態
  - 強すぎる制約
    - 制約の内容がシステムの仕様やビジネスルールに基づいて記述される状態

## 10章 転んだ後のバックアップ
- 論理バックアップ
  - SQLやCSVとしてDBそのものを再構成できるようにバックアップを取ること
    - mysqldumpやpg_dumpというコマンドで取れる
- 物理バックアップ
  - データベースの物理ファイルをまるごとバックアップする手法のこと
    - OSのcpやrsyncコマンドや専用ツールを使ったり
    - 再配置して再起動が必要になったりするので、データベースの停止も必要になることも
- PITR(ポイントインタイムリカバリ)
  - 特定の日時の状態にデータをリストアできる手法
    - バックアップファイルと更新情報の入ったログが必要
    - MySQLではバイナリログ、PostgreSQLではアーカイブログと呼ぶ
- 3つのバックアップにおける指針
  - RPO
    - いつ時点のデータを復旧するか（目標復旧時点）
  - RTO
    - どれくらいの時間で復旧できるか（目標復旧時間）
  - RLO
    - どこまで復旧するか（目標復旧レベル）
- 稼働率とバックアップ設計の指標

| 稼働率    | 年間停止時間 | 難易度 | 
| ---      | ---        | --- | 
| 90%      | 36.5日     | バックアップとリストアで十分 | 
| 99%      | 3.65日     | オンプレミス環境なら予備マシンが必要、大規模データならリストア所要時間を把握しておく必要がある | 
| 99.9%    | 8.7時間     | 法定停電への対応や24時間365日サポートなど、システム外の仕組み作りも必要 | 
| 99.99%   | 52分       | バックアップからリストだけでは難しい。コールドスタンバイなどが必要 | 
| 99.999%  | 5分        | 遅延レプリケーションなどの専用システムが必要 | 
| 99.9999% | 32秒       | 無停止サーバなどが必要になってくる。コストは非常に高くなる | 

- バックアップが正しく行われていることを毎回確認し、失敗した時に必ず気づけるようにする
- リストアを定期的に行う
- 手順書をまとめる。ベーシックな手順書だけではなく、実際のユースケースに則したパターンをいくつか用意すると良い
